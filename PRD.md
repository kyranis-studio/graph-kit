## 1. Overview

GraphKit is a lightweight, TypeScript-first library for building node graph applications in Deno 2. It provides the computational layer for creating, connecting, and executing node-based workflows, with primary focus on AI workflows while remaining general-purpose.

### 1.1 Vision

GraphKit enables developers to:

- **Model complex workflows** as node graphs with typed inputs and outputs
- **Execute graphs sequentially or conditionally** with configurable logging
- **Integrate AI models** seamlessly via pre-built nodes and streaming support
- **Debug workflows interactively** with step-through mode and real-time streaming display
- **Extend the system** via middleware, custom nodes, and custom storage backends

### 1.2 Inspiration

- **LangGraph**: State-based execution, conditional edges, cycles support
- **statelyai/graph**: Plain JSON graphs, port system for node-editor style graphs
- **@codemix/graph**: Type-safe graph database patterns
- **Ollama**: Local LLM inference patterns

## 3. Core Concepts

### 3.1 Node System

Nodes are typed execution units with unique identifiers. They define inputs, outputs, and the logic that processes data.

#### 3.1.1 Node Definition

```typescript
interface Node {
  id: string;
  type: string;
  inputs: Map<string, Port>;
  outputs: Map<string, Port>;
  data: Record<string, unknown>;
  metadata?: NodeMetadata;
  execute: NodeExecutor;
}

interface NodeMetadata {
  label?: string;
  description?: string;
  category?: string;
  color?: string;
  position?: { x: number; y: number };
}
```

### 3.2 Port System

Ports define the interface of a node — what data it accepts and produces.

```typescript
interface Port {
  id: string;
  name: string;
  type: PortType;
  required: boolean;
  defaultValue?: unknown;
  schema?: unknown;
}

type PortType = 'string' | 'number' | 'boolean' | 'object' | 'array' | 'any' | string;
```

- **Input Ports**: Receive data from incoming edges
- **Output Ports**: Send data to outgoing edges
- Support for required/optional ports
- TypeScript generics for type safety

### 3.3 Edge System

Edges connect output ports to input ports, carrying data between nodes. They support one-to-one and one-to-many connections.

```typescript
interface Edge {
  id: string;
  sourceNodeId: string;
  sourcePortId: string;
  targetNodeId: string;
  targetPortId: string;
  metadata?: EdgeMetadata;
}

interface EdgeMetadata {
  label?: string;
  color?: string;
  animated?: boolean;
}
```

#### 3.3.1 Connection Rules

- **Type Checking**: Output port type must be compatible with input port type
- **Cycle Prevention**: Optional, configurable cycle detection
- **Connection Limits**: Allow/disallow multiple connections to same input port (configurable)

### 3.4 Graph Structure

A graph contains nodes and edges. It's a plain JSON-serializable object that supports hierarchical/sub-graphs (optional, future).

```typescript
interface Graph {
  id: string;
  nodes: Map<string, Node>;
  edges: Map<string, Edge>;
  metadata?: GraphMetadata;
}
```

#### 3.4.1 Graph Operations

| Method | Description |
|--------|-----------|
| `addNode(type, config?)` | Create and add a node |
| `removeNode(nodeId)` | Remove node and connected edges |
| `addEdge(connection)` | Connect two ports |
| `removeEdge(edgeId)` | Remove an edge |
| `getNode(nodeId)` | Retrieve a node |
| `getEdgesForNode(nodeId)` | Get all edges connected to a node |
| `getPredecessors(nodeId)` | Get nodes that feed into a node |
| `getSuccessors(nodeId)` | Get nodes that receive data from a node |

## 4. Execution Engine

### 4.1 Node Execution Model

Nodes define an `execute` function that receives input values and returns output values. Supports async execution.

```typescript
type NodeExecutor<TInput = unknown, TOutput = unknown> = (
  inputs: TInput,
  context: ExecutionContext
) => Promise<TOutput> | TOutput;

interface ExecutionContext {
  graph: Graph;
  nodeId: string;
  state: GraphState;
  config?: Record<string, unknown>;
}
```

### 4.2 Graph Execution Modes

#### 4.2.1 Sequential Execution (Default)

Executes nodes in topological order.

- Wait for all inputs to be available
- Support for parallel execution of independent nodes
- Minimal logging enabled by default (`logLevel: 'minimal'`)
- Real-time LLM streaming display in `verbose` mode or chat node with `streaming: true`

```typescript
const engine = new ExecutionEngine({ logLevel: 'minimal' });
const result = await engine.execute(graph);
```

#### 4.2.2 Debug Execution Engine

`DebugExecutionEngine` is a standalone execution engine with rich CLI feedback,
lifecycle hooks, streaming support, and optional step-by-step mode.

- Step-by-step execution via `stepMode: true`
- Rich CLI feedback with colors, timing, and data previews

- Lifecycle hooks (`onNodeStart`, `onNodeComplete`, `onNodeError`, `onStreamChunk`)
- Comprehensive execution logging via `executionLog`

#### 4.2.3 Workflow Execution (State-based)

`Workflow` provides state-based execution with conditional routing, inspired by LangGraph.

- Support for `START` and `END` logic
- Conditional edges (routing based on state)
- Minimal logging enabled by default (`logLevel: 'minimal'`)
- Human-in-the-loop support via `pause`/`resume` (in planning)

```typescript
const workflow = graph.createWorkflow({
  startNode: 'start',
  endNode: 'end',
  logLevel: 'verbose',
  onStateUpdate: (state) => console.log('State updated'),
});
await workflow.run();
```

**Constructor Signature:**

#### 4.2.4 Web UI Execution Mode

The `WebUIExecutionEngine` provides a browser-based execution mode that serves a web page for visual graph execution and debugging.

- **Auto Mode**: Displays the graph as an interactive SVG visualization. Press "Execute" to run all nodes with real-time node status updates.
- **Debug Mode**: Splits the page into two panels — left panel shows the graph visualization with execution controls, right panel shows debug information (execution log, node inputs/outputs, streaming content, and timing).
- **Real-time updates**: Server-Sent Events (SSE) stream node lifecycle events to the browser
- **Step-by-step execution**: In debug mode, execution pauses before each node; press "Step" to continue
- **Node inspection**: Click on any node to view its ports, input/output values, streaming content, and errors
- **Workflow support**: When a graph has a configured workflow (startNode, endNode, conditional edges), the engine automatically uses state-based execution instead of topological sort:
  - **State-based routing**: Follows `startNode → endNode` path, evaluating conditional edges at runtime
  - **Cycle support**: Allows loops and repeated node execution (guarded by `maxSteps`)
  - **Workflow visualization**: Shows a collapsible group container around workflow nodes, START/END badges on nodes, and conditional edges rendered as dashed amber lines
  - **Conditional branch logging**: SSE event `workflowConditionEval` reports routing decisions in real-time

```typescript
const engine = new WebUIExecutionEngine({
  port: 3000,
  debugMode: true, // Split-panel UI with step control

const state = await engine.execute(graph);
```

### 4.3 Log Engine

The Log Engine is a centralized, extensible logging system that provides structured and styled output for graph execution and debugging. It supports three verbosity modes and uses a consistent color scheme for visual hierarchy.

#### 4.3.1 Log Level Configuration

The log engine operates in three modes, controlled via the `logLevel` configuration:

| Mode | Description | Output | Use Case |
|------|------|--------|----------|
| `'muted'` | No logging output | Suppressed — no console output, no side effects | Production, testing |
| `'minimal'` | Default mode | Displays essential execution progress: node start/completion, timestamps, and high-level status | Routine development |
| `'verbose'` | Detailed debugging | Shows full execution trace: node inputs/outputs, streaming content, timing, debug metadata, and intermediate state | Development, troubleshooting |

**Mapping to internal log levels:**

| Config Value | Internal Level | Description |
|-------------|----------------|-------------|
| `'muted'` | `silent` | No output |
| `'minimal'` | `info` | General information and status updates |
| `'verbose'` | `debug` + `trace` | Detailed debugging + maximum verbosity for streaming |

```typescript
// Muted mode — no logging
const mutedEngine = new ExecutionEngine({ logLevel: 'muted' });

// Minimal mode (default)
const minimalEngine = new ExecutionEngine({ logLevel: 'minimal' });

// Verbose mode — full debug output
const verboseEngine = new ExecutionEngine({ logLevel: 'verbose' });
```

**Log levels hierarchy:**

| Level | Description | Use Case |
|-------|-------|------|
| `silent` | No output | Production, testing |
| `error` | Critical failures only | Error monitoring |
| `warn` | Warnings and non-critical issues | Alert tracking |
| `info` | General information and status | Default for minimal mode |
| `debug` | Detailed debugging info | Development, troubleshooting |
| `trace` | Maximum verbosity — all data flow | Deep debugging, streaming |

#### 4.3.2 Background Color Hierarchy

The log engine uses different background colors to visually represent the hierarchy of log information. Each level of the hierarchy has a distinct background color, making it easy to scan and understand the structure at a glance. No outline characters are used — only background color blocks and spacing.

**Hierarchy levels and their background colors:**

| Level | Description | Background Color | Visual Effect |

**Visual formatting rules:**

1. **Graph boundaries**: Use `bgElevated` background with `textPrimary` text
2. **Node blocks**: Use `bgSurface` background for the entire node execution block
3. **Node details**: No background, use `textSecondary` for timestamps, `textPrimary` for status
4. **Streaming content**: Use `bgAccent` with 12.5% opacity as background
5. **Debug info**: Use `bgGray` with `textMuted` for non-essential information
6. **Spacing**: Use blank lines between major hierarchy levels instead of separator characters

#### 4.3.3 Minimal Mode

The **minimal** mode is the default logging level designed for production and routine development use. It provides essential execution visibility without overwhelming the developer with excessive detail.

**What it displays:**
- Graph execution start and completion markers with `bgElevated` background
- Sequential node execution with progress numbering (e.g., `[1/5]`)
- Node identifiers (name and label) with `bgSurface` background
- Node start timestamps with `textSecondary` color
- Node completion status with execution duration and `success` color
- Error indicators for failed nodes with `error` color

**What it omits:**
- Input/output port data (except streaming content when enabled)
- Debug metadata
- Intermediate state information
- Timing breakdowns

**Streaming condition in minimal mode:**
- Streaming content (thinking/response) is displayed **only if** the chat node has `streaming: true` set in its configuration
- If `streaming: false` or `streaming` is not specified, streaming content is omitted
- When streaming is enabled, thinking content is displayed with `accentHighlight` color and response content with `success` color

**Use cases:**
- Production deployments where verbose logging impacts performance
- Routine graph execution monitoring
- User-facing applications with execution progress feedback
- CI/CD pipelines and automated testing

```
Level 1: Graph-level events (start/end)
  [bgElevated] — Elevated surface background (#242427)
  
Level 2: Node-level events (start/complete/error)
  [bgSurface] — Standard surface background (#18181b)
  ├── Level 3: Port-level events (inputs/outputs)
  │   No background — clean text display
  └── Level 3: Timing information
      No background — clean text display
      └── Level 4: Streaming content (verbose only)
           [bgAccent-tint] — Subtle blue tint (#6282ff20)
  └── Level 3: Debug metadata (verbose only)
      [bgGray] — Muted background with textMuted
```

**Visual elements:**

| Element | Style | Color | Background | Context |
|---------|-----|-----|--------|---|
| Node start | `●` + number | `accent` (electric blue) | `bgSurface` | Node identifiers |
| Timestamp | `▸ started at` | `textSecondary` | None | Timing info |
| Inputs/Outputs | `▸ inputs:` / `▸ outputs:` | `textPrimary` | `bgSurface` | Data context |
| Thinking stream | `▸ thinking:` | `accentHighlight` (vivid purple) | `bgAccent-tint` | LLM reasoning |
| Response stream | `▸ response:` | `success` (vibrant green) | `bgAccent-tint` | LLM output |
| Completion | `✓` | `success` (vibrant green) | None | Successful nodes |
| Duration | `in Xms` / `in Xs` | `warning` (amber) | None | Execution timing |
| Errors | `✗` | `error` (bright red) | None | Failed nodes |
| Warnings | `⚠` | `warning` (amber) | None | Non-critical issues |
| Status headers | Plain text block | `textPrimary` | `bgElevated` | Section dividers |
| Graph state | Plain text block | `textPrimary` | `bgElevated` | Start/end markers |

**Note on visual markers:** The symbols (●, ✓, ✗, ⚠, ▸) are Unicode characters used for semantic clarity, not outline/box-drawing characters. They serve as status indicators within the text content, not as structural boundaries. Background colors provide the visual grouping.

#### 4.3.6 Log Engine Architecture

The log engine follows a composable, overrideable architecture:

```typescript
interface LogEngineConfig {
  logLevel: 'muted' | 'minimal' | 'verbose';
  logger?: LogLogger;        // Custom logger implementation
  formatter?: LogFormatter;  // Custom output formatter
  timestamp?: boolean;       // Enable/disable timestamps
}

abstract class LogEngine {
  protected config: LogEngineConfig;
  
  // Core logging methods
  abstract log(level: LogLevel, message: LogMessage): void;
  abstract logNodeStart(node: NodeInfo): void;
  abstract logNodeComplete(node: NodeInfo, duration: number): void;
  abstract logNodeError(node: NodeInfo, error: Error): void;
  abstract logGraphStart(graph: GraphInfo): void;
  abstract logGraphComplete(success: boolean): void;
  abstract logStream(content: string, type: 'thinking' | 'response'): void;
  abstract logDebug(key: string, value: unknown): void;
  
  // Utility methods
  protected formatMessage(message: LogMessage): string;
  protected applyColors(text: string, color: ColorKey): string;
  protected shouldLog(level: LogLevel): boolean;
}

// Color keys for consistent theming
type ColorKey = 
  | 'accent' | 'accentHighlight' | 'success' | 'warning' | 'error'
  | 'info' | 'textPrimary' | 'textSecondary' | 'textMuted'
  | 'bgSurface' | 'bgElevated' | 'bgGray' | 'bgSuccess' | 'bgError' | 'bgAccent';
```

```typescript
  }
```

## 5. Debug Execution Engine

The `DebugExecutionEngine` is a standalone execution engine with rich CLI
feedback, lifecycle hooks, streaming support, and optional step-by-step mode. It
runs nodes in topological order, displaying real-time execution progress with
color-coded output for inputs, outputs, timing, and streaming content.

### 5.1 Configuration

The engine accepts an optional config object — there is no separate config
interface; all options are passed inline:

```typescript
class DebugExecutionEngine {
  constructor(config?: {
    stepMode?: boolean;          // Pause between nodes (default: false)
    forceFullLog?: boolean;      // Force verbose logging (default: false)
    forceStreaming?: boolean;    // Force streaming on chat nodes (default: false)
    logLevel?: 'silent' | 'muted' | 'minimal' | 'verbose';
    onNodeStart?: (info: NodeDebugInfo) => void;
    onNodeComplete?: (info: NodeDebugInfo) => void;
    onNodeError?: (info: NodeDebugInfo) => void;
    onStreamChunk?: (state: DebugStreamState) => void;
  });
}
```

- **`stepMode`**: When `true`, execution pauses before each node. Press
  `SPACE` to continue or `ESC` to cancel.
- **`forceFullLog`**: When `true`, sets `logLevel` to `'verbose'` for full
  execution trace including all node inputs/outputs.
- **`forceStreaming`**: When `true`, overrides `streaming: false` on all chat
  nodes to enable real-time streaming display.
- **`logLevel`**: Log verbosity (`'silent'`, `'muted'`, `'minimal'`,
  `'verbose'`). Defaults to `'minimal'` unless `forceFullLog` is `true`.

### 5.2 Lifecycle Hooks

The engine provides four optional hooks for custom instrumentation:

```typescript
const debugEngine = new DebugExecutionEngine({
  onNodeStart: (info) => console.log(`Starting: ${info.nodeId}`),
  onNodeComplete: (info) => console.log(`Done: ${info.nodeId} (${info.duration?.toFixed(0)}ms)`),
  onNodeError: (info) => console.error(`Failed: ${info.nodeId}`, info.error),
  onStreamChunk: (chunk) => process.stdout.write(chunk.state.response),
});
```

Each hook receives a `NodeDebugInfo` or `DebugStreamState` object with full
execution metadata (node ID, type, inputs, outputs, duration, status, and
predecessor/successor lists).

### 5.3 Step-by-Step Execution

Enabling `stepMode` pauses the engine before each node and waits for keyboard
input:

| Key | Action |
|-----|--------|
| `SPACE` | Execute the next node |
| `ESC` | Cancel entire execution |

After cancelling, the engine stops processing further nodes and the graph
state contains partial results from completed nodes. Check `engine.isCancelled`
to detect cancellation.

```typescript
const debugEngine = new DebugExecutionEngine({ stepMode: true });
const state = await debugEngine.execute(graph);
```

### 5.4 Public API

```typescript
const engine = new DebugExecutionEngine(config);

// Execute the graph (runs nodes in topological order)
const state: GraphState = await engine.execute(graph, initialState?);

// Cancel a running execution
engine.cancel();

// Reset execution log and cancellation state
engine.reset();

// Access execution log after run
const log: ReadonlyArray<NodeDebugInfo> = engine.executionLog;

// Check cancellation status
const cancelled: boolean = engine.isCancelled;
```

The `GraphState` returned by `execute()` contains a `values` Map
(`Map<string, unknown>`) keyed by `${nodeId}.${portId}` and an optional
`messages` array for AI conversation history.

### 5.5 Web UI Server Mode

The `WebUIExecutionEngine` provides a browser-based alternative to the CLI debug
engine. It starts an HTTP server that serves a web page with an interactive SVG
graph visualization, real-time execution updates via Server-Sent Events (SSE),
and a split-panel debug interface in debug mode.

#### 5.5.1 Overview

The Web UI mode runs the same graph execution engine under the hood but replaces
the terminal output with a browser-based experience:

- **Auto Mode**: Displays the graph as nodes with edges. Click "Execute" to run
  all nodes in topological order. Nodes change color in real-time (gray →
  blue → green/red) as they execute.
- **Debug Mode**: The page splits horizontally — left panel shows the graph
  visualization with step controls, right panel shows a structured debug log
  with inputs, outputs, streaming content, and timing.

#### 5.5.2 Architecture

```
│  │   Graph Panel        │   Debug Panel (debug     │ │
│  │   - SVG nodes/edges  │   mode only)             │ │
│  │   - Color status     │   - Execution log        │ │

│             │                    │                    │

│        │  REST   │         │   SSE   │               │
│        │  API    │         │  Events │               │
```

#### 5.5.3 REST API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/graph` | Returns the graph definition (nodes, edges, metadata) |
| `POST` | `/api/execute` | Starts graph execution |
| `POST` | `/api/step` | Advances execution by one node (debug mode) |
| `POST` | `/api/cancel` | Cancels running execution |
| `GET` | `/api/events` | SSE stream for real-time execution events |

#### 5.5.4 SSE Event Stream

The server pushes Server-Sent Events to the browser for every execution
lifecycle event:

| Event | Payload | Description |
|-------|---------|-------------|
| `executionStart` | `{ totalNodes, nodeOrder, workflowStartNode?, workflowEndNode? }` | Execution has begun (includes workflow markers when applicable) |
| `nodeStart` | `{ nodeId, nodeType, label, inputs, index, total, predecessors, successors, isConditionalSource? }` | A node is about to execute |
| `nodeComplete` | `{ nodeId, duration, outputs }` | A node finished successfully |
| `nodeError` | `{ nodeId, error }` | A node failed |
| `nodeSkipped` | `{ nodeId }` | Node was skipped (cancellation) |
| `executionPaused` | `{ nodeId }` | Execution paused before node (debug mode) |
| `executionResumed` | `{ nodeId }` | Execution resumed after step |
| `streamChunk` | `{ nodeId, state: { response, thinking, done } }` | LLM streaming content |
| `workflowConditionEval` | `{ sourceNodeId, targetNodeId }` | A workflow conditional edge was evaluated and routing decision made |
| `graphComplete` | `{ success, summary, totalNodes, steps? }` | Execution finished (includes step count for workflows) |
| `executionError` | `{ error }` | Fatal execution error |

## 8. Advanced Features

### 8.1 Conditional Edges

Workflows support conditional routing via `addConditionalEdge()`, which takes a
`sourceNodeId` and a `condition` function. The condition receives the current
`GraphState` and returns the `nodeId` of the next node to execute. This enables
dynamic branching and loops.

```typescript
workflow.addConditionalEdge({
  sourceNodeId: 'router',
  conditionLabel: 'positive ? end-positive : end-negative', // optional label for visualization
  condition: (state) => {
    const val = state.values.get('router.output') as number;
    return val > 0 ? 'end-positive' : 'end-negative';
  },
});
```

Conditional edges are stored in the graph's `workflowConfig` when added via the
workflow API, making them accessible to visualization tools like the Web UI.

### 8.2 Subgraphs

Subgraphs (nested groups of nodes with their own boundaries) are not yet
implemented as a first-class concept in the core library. However, the Web UI
frontend supports **visual groups** via dagre compound graphs — workflow nodes
are automatically rendered inside a collapsible group container when a workflow
is configured on the graph.

Export graphs to Mermaid and DOT formats for diagram generation. Built-in support for node colors and labels.

### 9.6 Interactive Chat Node

The `interactive-chat` node provides a dual-mode terminal chat node: it can run as an **interactive REPL** (reading from stdin) or as a **workflow node** (receiving `userMessage` via edges from other nodes). It supports multiple AI providers with streaming, tool-calling, and session-persisted conversation history.

**Key features:**
- **Dual mode**: Interactive REPL mode for direct terminal use, and workflow mode for graph-based pipelines
- **Multi-provider**: Supports Ollama, OpenAI-compatible, and OpenRouter providers
- **Tool-calling loop**: Automatic multi-round function calling with tool result feedback (max 10 turns)
- **Streaming support**: Real-time display of thinking and response content (both modes)
- **Session persistence**: Conversation history stored per `sessionId` across executions
- **Configurable**: Provider, model, system prompt, temperature, API keys, streaming toggle
- **Debug mode compatible**: Works with `DebugExecutionEngine` step-by-step execution

**Node interface:**

```typescript
graph.registerNodeType('interactive-chat', getInteractiveChatNodeType());

// Inputs:
// - provider: string (default: 'ollama') — AI provider: 'ollama', 'openai', or 'openrouter'
// - model: string (default: 'llama3') — LLM model name
// - systemPrompt: string (optional) — System prompt for the conversation
// - temperature: number (default: 0.7) — LLM temperature
// - streaming: boolean (default: true) — Enable streaming output
// - userMessage: string (optional) — When provided, the node runs in workflow mode
//   (single-turn send + response) instead of interactive REPL mode
// - initialPrompt: string (optional) — First message to auto-send in REPL mode
// - apiKey: string (optional) — API key for OpenAI or OpenRouter providers
// - baseUrl: string (optional) — Custom base URL for the provider
// - tools: InteractiveTool[] (optional) — Tool definitions and handlers for
//   tool-calling. When provided, the node runs a tool-calling loop instead of
//   streaming: the model may request tool invocations, results are fed back,
//   and the loop continues until a text response is returned (max 10 turns).
// - sessionId: string (optional, default: nodeId) — Conversation session key
//   for persisting message history across workflow executions

// Outputs:
// - response: string — Last AI response in the conversation
// - conversation: array — Full conversation history
// - tokenCount: number — Total tokens used
```

**Workflow mode vs Interactive mode:**

The node automatically selects its mode based on the `userMessage` input:

- **Workflow mode** (when `userMessage` is provided via an edge): Sends the message
  to the LLM, handles tool-calling loops if tools are configured, captures streaming
  output, and returns the final response. No terminal interaction occurs.
- **Interactive REPL mode** (when `userMessage` is absent): Enters a terminal
  read-eval-print loop that reads from stdin, supports `initialPrompt`,
  `exit`/`quit`/`q` commands, and displays tool calls inline.

**Session persistence:**

Conversation history is stored in the graph state under the key
`__interactive_chat_{sessionId}`. This allows multi-turn conversations to persist
across workflow cycles (e.g., in a loop that routes back to the same node). Each
node can have its own `sessionId` or share one with other nodes for shared context.

**Terminal output example during execution (REPL mode):**

```
  ┌────── Interactive Chat ──────┐
  │ Model: llama3.2
  └──────────────────────────────┘
  type exit, quit, or q to end

  You: Hello
  AI: Hello! How can I help you today?

  You: Explain node graphs
  AI: A node graph is a data structure...
```

**User controls:**
- Type a message and press Enter to send
- Type `exit`, `quit`, or `q` to end the session
- The conversation history is returned in the `conversation` output

**Integration with Debug Engine:**

When used with `DebugExecutionEngine`, the step mode pauses before the interactive chat node starts, allowing the user to step into the interactive session. Once inside, the node takes over stdin for the conversation loop. After the user exits, control returns to the debug engine for subsequent nodes.

```
// Press SPACE to start interactive chat
[PAUSED] Press SPACE to continue, ESC to cancel
// ...user presses space...
// Interactive chat session begins
You: Hello
AI: Hello! How can I help?
You: exit
// Interactive session ends
// Debug engine continues with next node
```

```typescript
    streaming: true,
    temperature: 0.7,
  },
});

graph.on('llmStreamChunk', ({ nodeId, state }) => {
  if (state.thinking) {
    console.log('Thinking:', state.thinking);
  }
  if (state.done) {
    console.log('Final Response:', state.response);
  }
});

const result = await graph.execute();
console.log('Response:', result.values.get(`${chatNode.id}.response`));
const thinking = result.values.get(`${chatNode.id}.thinking`);
if (thinking) {
  console.log('Thinking:', thinking);
}
```

### 11.4 Function Calling

```typescript
import { createOllamaProvider } from '../ai/providers/ollama.ts';
import type { ToolDefinition, ChatMessage } from '../ai/providers/types.ts';

const weatherTool: ToolDefinition = {
  type: 'function',
  function: {
    name: 'get_weather',
    description: 'Get the current weather for a location',
    parameters: {
      type: 'object',
      properties: { location: { type: 'string' } },
      required: ['location'],
    },
  },
};

async function getWeather(location: string): Promise<string> {
  return `The weather in ${location} is 22°C and sunny.`;
}

const ollama = createOllamaProvider();
const messages: ChatMessage[] = [
  { role: 'user', content: "What's the weather in Tokyo?" },
];

let iterations = 0;
while (iterations < 5) {
  iterations++;
  const response = await ollama.chat({
    model: 'llama3.1',
    messages,
    tools: [weatherTool],
  });

  if (response.message.tool_calls) {
    messages.push(response.message);
    for (const tc of response.message.tool_calls) {
      const args = JSON.parse(tc.function.arguments);
      const result = await getWeather(args.location);
      messages.push({ role: 'tool', content: result, tool_call_id: tc.id });
    }
  } else {
    console.log('Final:', response.message.content);
    break;
  }
}
```

### 11.8 Interactive Chat
